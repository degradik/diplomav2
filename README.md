# Веб-приложение для регистрации участников и управления мероприятиями

## 1. Описание проекта
Проект: веб-приложение для регистрации пользователей и управления мероприятиями.

Ключевой пользовательский сценарий:
1. Пользователь заходит в систему.
2. Регистрируется, заполняя: ФИО, телефон, email, УЗ (учебное заведение) из списка.
3. Авторизуется и видит список доступных мероприятий.
4. Регистрируется на мероприятие и может отменить регистрацию.

Ключевой административный сценарий:
1. Администратор входит в отдельную админ-зону.
2. Создает, редактирует, публикует и архивирует мероприятия.
3. Управляет списком УЗ, пользователями и участниками мероприятий.

## 2. Цели и границы MVP
### Цели MVP
- Реализовать стабильную регистрацию и авторизацию пользователей.
- Реализовать личный кабинет с отображением доступных мероприятий.
- Реализовать запись/отмену записи на мероприятие.
- Реализовать отдельную админку для управления мероприятиями.

### Что входит в MVP
- Роли `USER` и `ADMIN`.
- Справочник УЗ (Institution).
- Базовые CRUD-операции для мероприятий в админке.
- Журнал регистраций пользователей на мероприятия.

### Что можно вынести в пост-MVP
- Email/SMS уведомления.
- Экспорт отчетов (Excel/PDF).
- QR-check-in на площадке.
- Интеграции с внешними календарями.

## 3. Пользователи и роли
### Роль USER (участник)
- Регистрация и вход.
- Просмотр профиля и редактирование контактных данных.
- Просмотр доступных мероприятий.
- Регистрация на мероприятие.
- Отмена своей регистрации.

### Роль ADMIN (администратор, отдельная сущность доступа)
- Вход в `/admin`.
- Управление УЗ.
- Создание/редактирование/удаление/архивация мероприятий.
- Управление участниками мероприятия (просмотр, удаление из списка участников).
- Просмотр базовой аналитики: количество регистраций, загрузка мероприятий.

## 4. Функциональные требования
### 4.1 Регистрация пользователя
Поля формы:
- `fullName` (ФИО) — обязательно, 5-120 символов.
- `phone` — обязательно, валидация формата.
- `email` — обязательно, уникально, валидация формата.
- `password` — обязательно, минимум 8 символов.
- `institutionId` (УЗ из списка) — обязательно.

Проверки:
- Нельзя создать пользователя с существующим email.
- Нельзя выбрать несуществующее УЗ.
- Ошибки валидации показываются рядом с полем.

### 4.2 Авторизация
- Вход по `email + password`.
- После входа выдается JWT.
- JWT хранится безопасно (рекомендован `httpOnly cookie`; допустимо `localStorage` для учебного MVP).

### 4.3 Профиль пользователя
- Отображение ФИО, телефона, email, УЗ.
- Редактирование телефона и УЗ.
- Email изменяется через отдельный подтверждаемый сценарий (post-MVP) или недоступен к изменению в MVP.

### 4.4 Просмотр мероприятий пользователем
Пользователь видит:
- Название, описание, дата/время начала и окончания.
- Доступность мероприятия (доступно/недоступно).
- Статус своей регистрации (зарегистрирован/не зарегистрирован).

Правила доступности:
- Публичные мероприятия (`isPublic=true`) доступны всем.
- Непубличные доступны только пользователям выбранного УЗ (по бизнес-правилу).
- Мероприятия в прошлом недоступны для новой регистрации.

### 4.5 Регистрация на мероприятие
- Пользователь может зарегистрироваться, если:
  - мероприятие доступно по правилам;
  - текущая дата меньше `startAt`;
  - нет существующей регистрации.
- При успешной регистрации создается запись в `UserEvent`.

### 4.6 Отмена регистрации
- Пользователь может отменить только свою регистрацию.
- После отмены запись `UserEvent` удаляется.

### 4.7 Админка: УЗ
- Создание УЗ.
- Редактирование названия УЗ.
- Удаление УЗ с проверкой зависимостей.

### 4.8 Админка: мероприятия
Поля мероприятия:
- Название.
- Описание (контент).
- Дата/время начала и окончания.
- Привязка к УЗ (или флаг публичности).
- Признак публикации/архивации.

Операции:
- Создать мероприятие.
- Редактировать мероприятие.
- Удалить/архивировать мероприятие.
- Просмотреть список участников.

Валидации:
- `startAt < endAt`.
- Нельзя создать событие с пустым названием.
- Нельзя публиковать событие с некорректными датами.

### 4.9 Админка: участники
- Просмотр списка участников по мероприятию.
- Фильтрация по ФИО, email, УЗ.
- Удаление участника из конкретного мероприятия.

## 5. Нефункциональные требования
- Время ответа API: до 500 мс для типовых операций (локально).
- Безопасность: пароли только в хешированном виде.
- Логи ошибок на backend.
- Адаптивный frontend (desktop/tablet/mobile).
- Базовая доступность: читаемые формы, подписи, понятные ошибки.

## 6. Архитектура решения
### 6.1 Технологический стек
- Frontend: Vue 3 + Vite + Vue Router + Axios.
- Backend: Node.js + Express.
- ORM: Prisma.
- БД: SQLite (MVP), возможна миграция на PostgreSQL.
- Аутентификация: JWT.

### 6.2 Логическая схема модулей
- `Auth Module`: регистрация, вход, проверка токена.
- `User Module`: профиль пользователя.
- `Institution Module`: справочник УЗ.
- `Event Module`: события и фильтрация доступных событий.
- `Registration Module`: запись на событие.
- `Admin Module`: управление сущностями и участниками.

### 6.3 Маршрутизация
- Пользовательская зона: `/`, `/login`, `/register`, `/profile`, `/events`.
- Админ-зона: `/admin`, `/admin/events`, `/admin/institutions`, `/admin/users`.

## 7. Модель данных (целевая)
Базовые сущности:
- `Institution` (`id`, `name`).
- `User` (`id`, `name`, `email`, `passwordHash`, `phone`, `role`, `institutionId`, `createdAt`, `updatedAt`).
- `Event` (`id`, `title`, `content`, `startAt`, `endAt`, `institutionId`, `isPublic`, `status`, `createdAt`, `updatedAt`).
- `UserEvent` (`userId`, `eventId`, `registeredAt`).

Рекомендованные расширения:
- `Event.status`: `DRAFT | PUBLISHED | ARCHIVED`.
- Индексы: `Event(startAt)`, `Event(institutionId)`, `User(email unique)`, `UserEvent(eventId)`.

## 8. API-контракт (черновик)
### 8.1 Auth
- `POST /api/auth/register` — регистрация.
- `POST /api/auth/login` — вход.
- `GET /api/auth/me` — текущий пользователь.

### 8.2 User
- `GET /api/users/me` — профиль.
- `PATCH /api/users/me` — обновление профиля.

### 8.3 Institutions
- `GET /api/institutions` — список УЗ.
- `POST /api/admin/institutions` — создать УЗ (`ADMIN`).
- `PATCH /api/admin/institutions/:id` — изменить УЗ (`ADMIN`).
- `DELETE /api/admin/institutions/:id` — удалить УЗ (`ADMIN`).

### 8.4 Events
- `GET /api/events` — список доступных пользователю событий.
- `GET /api/events/:id` — карточка события.
- `POST /api/admin/events` — создать событие (`ADMIN`).
- `PATCH /api/admin/events/:id` — редактировать событие (`ADMIN`).
- `DELETE /api/admin/events/:id` — удалить/архивировать (`ADMIN`).

### 8.5 Registrations
- `POST /api/events/:id/register` — запись пользователя.
- `DELETE /api/events/:id/register` — отмена записи.
- `GET /api/admin/events/:id/participants` — список участников (`ADMIN`).

## 10. Безопасность
- Хеширование паролей (`bcrypt`, salt rounds >= 10).
- Проверка прав доступа middleware-уровнем.
- Валидация входных данных (`zod`/`joi`/аналог).
- Ограничение частоты запросов на auth-эндпоинтах.
- CORS только для доверенных origin.
- Сокрытие деталей внутренних ошибок.

## 11. Логирование и мониторинг
- Логи backend-запросов и ошибок.
- Отдельный лог security-событий (неуспешные входы, доступ к admin без прав).
- Метрики для админа (количество регистраций по событиям).

## 14. Тест-план
### Критические позитивные кейсы
- Пользователь успешно регистрируется.
- Пользователь успешно входит.
- Пользователь видит доступные мероприятия.
- Пользователь успешно записывается и отменяет запись.
- Админ создает мероприятие.

### Критические негативные кейсы
- Регистрация с уже существующим email.
- Неверный пароль при входе.
- Запись на прошедшее мероприятие.
- Попытка USER вызвать admin endpoint.
- Некорректные даты мероприятия (`startAt >= endAt`).

### Регрессия
- После изменений в админке пользовательская часть не ломается.
- После изменений auth не ломается доступ к уже существующим сценариям.

## 15. Риски и меры
- Риск: слабая валидация данных.
  - Мера: строгие DTO и единый validation middleware.
- Риск: утечка прав в админку.
  - Мера: централизованный RBAC middleware + тесты прав доступа.
- Риск: дублирующиеся регистрации.
  - Мера: составной ключ `userId + eventId` и проверка в сервисе.
- Риск: рост нагрузки.
  - Мера: индексы, пагинация, миграция с SQLite на PostgreSQL.
