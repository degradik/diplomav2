# Веб-приложение для регистрации участников и управления мероприятиями

## 1. Тема и назначение
**Тема ВКР:** разработка веб-приложения для регистрации участников и управления мероприятиями.

Проект решает две основные задачи:
1. Дать обычному пользователю удобный интерфейс регистрации, выбора учебного заведения, просмотра и записи на мероприятия.
2. Дать администратору отдельную защищенную админ-панель для управления мероприятиями, учебными заведениями и пользователями.

Репозиторий можно использовать как:
1. Рабочий код проекта.
2. Актуальный план написания диплома (аналитика, проектирование, реализация, тестирование, внедрение).

## 2. Актуальное состояние проекта (на сегодня)

### 2.1 Что уже реализовано
**Backend (Node.js + Express + Prisma + SQLite):**
1. Регистрация, авторизация, получение текущего пользователя (`/api/auth/register`, `/api/auth/login`, `/api/auth/me`).
2. Роли `USER` и `ADMIN`, защита маршрутов (`authRequired`, `adminOnly`).
3. Профиль пользователя (`/api/users/me`) с обновлением телефона, учебного заведения и аватара.
4. События для пользователей:
- список доступных мероприятий (`/api/events`);
- детальная страница (`/api/events/:id`);
- запись и отмена записи (`POST/DELETE /api/events/:id/register`).
5. Админ API:
- dashboard-метрики (`/api/admin/dashboard`);
- управление учебными заведениями (`POST/PATCH/DELETE /api/admin/institutions/:id`);
- управление пользователями (создание + список) (`POST/GET /api/admin/users`);
- управление мероприятиями (`GET/POST/PATCH/DELETE /api/admin/events`);
- участники мероприятия (`GET/DELETE /api/admin/events/:id/participants...`).
6. Валидация входных данных через `zod`.
7. Базовые меры безопасности: `helmet`, `cors`, rate-limit, `bcrypt`, санитизация HTML-контента мероприятия (`sanitize-html`), лимиты размеров полезной нагрузки.

**Frontend (Vue 3 + Vite + Vue Router):**
1. Пользовательские страницы:
- `/login`, `/register`;
- `/profile`;
- `/events`, `/events/:id`.
2. Админские страницы:
- `/admin` (вход);
- `/admin/events`;
- `/admin/institutions`;
- `/admin/users`.
3. Единый дизайн-системный слой UI-компонентов (кнопки, поля, табы, карточки, уведомления).
4. Обновленная админка в едином стиле (по аналогии Moonshine):
- разделы с hero-блоком, статистикой, рабочими панелями и реестрами;
- отдельные переиспользуемые компоненты для `events`, `institutions`, `users`.
5. На странице мероприятий пользователя:
- только доступные события (публичные или для его учебного заведения);
- русские статусы;
- карточки и детальная страница мероприятия.

### 2.2 Что частично реализовано / требует доработки
1. В админке пользователей реализованы создание и просмотр, но нет отдельного UI для редактирования/удаления пользователя.
2. Автоматизированные тесты (unit/integration/e2e) пока не оформлены как отдельный тестовый пакет.
3. WYSIWYG-редактор отключен в текущем UI админки мероприятий в пользу стабильного `textarea` (longtext).

## 4. Архитектура и модель данных

### 4.1 Основные сущности
1. `Institution` — учебные заведения.
2. `User` — пользователи (роль, контакты, аватар, привязка к учебному заведению).
3. `Event` — мероприятия (статус, доступность, даты, описание, изображение).
4. `UserEvent` — факт регистрации пользователя на мероприятие (many-to-many).

### 4.2 Текущая структура БД (Prisma)
```prisma
model Institution {
  id     Int    @id @default(autoincrement())
  name   String @unique
  users  User[]
  events Event[]
}

model User {
  id            Int          @id @default(autoincrement())
  name          String
  email         String       @unique
  passwordHash  String       @map("password")
  role          String       @default("USER")
  avatarUrl     String?
  phone         String?
  institutionId Int?
  institution   Institution? @relation(fields: [institutionId], references: [id], onDelete: SetNull)
  registrations UserEvent[]
  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @default(now()) @updatedAt
}

model Event {
  id            Int          @id @default(autoincrement())
  title         String
  imageUrl      String?
  startAt       DateTime
  endAt         DateTime
  institutionId Int?
  isPublic      Boolean      @default(true)
  status        String       @default("DRAFT")
  content       String
  institution   Institution? @relation(fields: [institutionId], references: [id], onDelete: SetNull)
  participants  UserEvent[]
  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @default(now()) @updatedAt

  @@index([institutionId])
  @@index([startAt])
  @@index([status])
}

model UserEvent {
  userId       Int
  eventId      Int
  registeredAt DateTime @default(now())
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  event        Event    @relation(fields: [eventId], references: [id], onDelete: Cascade)

  @@id([userId, eventId])
  @@index([eventId])
}
```

## 5. Пользовательские и административные сценарии

### 5.1 Пользователь
1. Регистрация по ФИО, email, паролю.
2. Авторизация.
3. Заполнение профиля: телефон, учебное заведение, аватар.
4. Просмотр доступных мероприятий.
5. Переход в детальную страницу мероприятия.
6. Запись/отмена записи.

### 5.2 Администратор
1. Вход в `/admin`.
2. Управление мероприятиями:
- создание;
- редактирование;
- архивирование;
- восстановление из архива;
- удаление;
- просмотр/удаление участников.
3. Управление учебными заведениями:
- создание;
- редактирование;
- удаление с проверкой зависимостей.
4. Управление пользователями:
- создание;
- просмотр и фильтрация.

## 7. Безопасность и надежность

Реализовано:
1. Хеширование паролей через `bcrypt`.
2. JWT-аутентификация и ролевой контроль доступа.
3. Ограничение частоты запросов (`auth limiter`, `global api limiter`, `write limiter`).
4. Валидация всех основных DTO через `zod`.
5. Санитизация HTML-контента мероприятий (`sanitize-html`).
6. Ограничение размера JSON (`2mb`) и ограничение размера картинок в `data:image`.
7. CORS по белому списку источников.
8. Централизованный обработчик ошибок.

## 8. Как запустить проект

### 8.1 Предусловия
1. `Node.js 18+`
2. `npm 9+`

### 8.2 Быстрый старт
```bash
npm run setup
npm run dev
```

После запуска:
1. Frontend: `http://localhost:5173`
2. Backend: `http://localhost:3000`
3. Health: `http://localhost:3000/api/health`

### 8.3 Ручной запуск по частям
```bash
npm run dev:backend
npm run dev:frontend
```

### 8.4 Переменные окружения backend
Файл `backend/.env`:
```env
DATABASE_URL="file:./dev.db"
JWT_SECRET="change-me"
CORS_ORIGINS="http://localhost:5173,http://127.0.0.1:5173"
```

## 12. Критерии готовности дипломного MVP

Проект можно считать готовым к защите, когда:
1. Пользовательский поток работает от регистрации до записи на мероприятие и отмены записи.
2. Админ-поток закрывает управление мероприятиями, учебными заведениями и пользователями.
3. Безопасность и валидация покрывают основные риски некорректного ввода.
4. README, инструкции запуска и демонстрационные данные полностью актуальны.
5. Подготовлены материалы для главы тестирования (таблицы кейсов и результаты прогонов).
